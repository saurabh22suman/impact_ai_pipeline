AI Impact Pipeline — Backend/API Developer Guide

Scope: backend + API runtime, execution flow, assumptions, invariants, and change safety.

1) Purpose and Audience

This guide is for backend/API developers working on the Go runtime. It explains how the codebase connects end-to-end: config loading, ingestion, enrichment, impact row generation, persistence, export, and API endpoints.

It is intentionally practical: where to edit, what assumptions exist, and what to re-verify before shipping changes.

2) Runtime Architecture (System-first view)

Main components

  • Frontend (nginx/static): UI and API proxy (see frontend/).
  • API process: cmd/api/main.go owns HTTP endpoints and request orchestration.
  • Engine service: internal/engine/service.go owns pipeline execution and exports.
  • Ingest fetchers/router: internal/ingest/router.go chooses RSS/direct/pulse strategy.
  • Enrichment + provider routing: internal/enrich/enricher.go and router/providers.
  • Storage abstraction: internal/storage/env.go picks file/memory/persistent mode.

Boot flow

  1. engine.Bootstrap() loads config and storage (internal/engine/bootstrap.go:25).
  2. Creates Service with config + store (internal/engine/bootstrap.go:44).
  3. Constructs fetchers and RouterFetcher (internal/engine/bootstrap.go:46-49).
  4. API server wires handlers in cmd/api/main.go:46-53.

3) End-to-End Execution Path

Run request path

POST /v1/runs in cmd/api/main.go:196 performs:

  1. Decode core.RunRequest and fill defaults (cmd/api/main.go:203-211).
  2. Resolve selected sources (engine.ResolveSources, call at cmd/api/main.go:213).
  3. Collect articles via fetcher router (cmd/api/main.go:219).
  4. Execute pipeline via Service.Run (cmd/api/main.go:225).
  5. Return run result + notices.

Pipeline core inside Service.Run

  1. Profile lookup (internal/engine/service.go:51).
  2. Dedupe ingested articles (internal/engine/service.go:67).
  3. Entity selection + impact mode setup (internal/engine/service.go:71).
  4. For each article:
    • Relevance score and threshold gate (internal/engine/service.go:76-79).
    • Enrichment (internal/engine/service.go:81).
    • NSE session alignment (internal/engine/service.go:86-91).
    • Feature row generation (internal/engine/service.go:93).
  5. Aggregate costs/tokens and artifact counts (internal/engine/service.go:108-118).
  6. Persist run/events/feature rows/artifacts (internal/engine/service.go:120-131).

4) Configuration Model and Assumptions

Config loading

config.Load loads multiple YAML files and validates invariants (internal/config/loader.go:13-66).

  • sources.yaml, entities.niftyit.yaml, entities.custom.yaml
  • optional entity_groups.yaml
  • factors.yaml, providers.yaml, pipelines.yaml

Important invariants (enforced)

  • At least one enabled source/provider/profile/factor (loader.go:166-212).
  • Default profile must exist (loader.go:213-215).
  • Fallback chain entries must point to enabled provider+model (loader.go:217-225).
  • Entity groups require valid parent/child roles (loader.go:187-206).
Assumption: config validation is strict. Most “weird runtime behavior” after config edits comes from invalid YAML semantics, not pipeline logic.

5) Ingest, Relevance, Enrich, Impact, Export (Core internals)

Ingest router

RouterFetcher.FetchWithNotices branches by normalized source kind and can fallback from RSS to direct when crawl_fallback=true (internal/ingest/router.go:34-61).

Relevance + LLM refinement trigger

Relevance score is computed first and compared to profile thresholds. Enricher sets needsLLM using NeedsLLMRefinement(score, noveltyThreshold, ambiguityThreshold) (internal/enrich/enricher.go:44).

LLM/provider path is conditional. If not triggered (or router errors), deterministic/rules output remains.

Entity matching semantics

Entity matching uses mapper + matcher and only considers enabled entities. This affects relevance and downstream event selection behavior.

Impact mode (parent-child)

Entity selection builds an impactModeConfig from requested entities and entity groups (internal/engine/service.go:259). Parent-child feature rows are built in internal/engine/impact_rows.go.

In impact mode, parent/child mapping constraints determine which rows are emitted and how ParentEntity/ChildEntity are assigned.

Export behavior

  • ExportJSONL requires non-empty events (internal/engine/service.go:135-141).
  • ExportCSV emits business-facing feature columns via exporter CSV (internal/engine/service.go:143-150, internal/export/exporter.go:32).
  • ExportTOON requires non-empty events (internal/engine/service.go:152-158).

7) Storage and Artifact Persistence

Store selection

NewStoreFromEnv chooses by STORAGE_MODE: memory, file, or persistent (internal/storage/env.go:12-73).

File mode layout

  runs/

Persistence writes are driven by persistRunArtifacts (internal/engine/service.go:168-200) and file store methods (internal/storage/file_store.go).

8) Key Assumptions and Common Pitfalls

  • Unknown requested entities are treated as invalid selection (400 path from API).
  • No-events runs can surface as errors in exports and some tests if fixtures are too weak.
  • Provider routing is threshold-dependent; deterministic output is expected in many paths.
  • Impact-mode selection can filter out rows if parent-child constraints are not satisfied.
  • Frontend/API tests may assert exact form/control behavior; UI contract changes must update tests.
When debugging “unexpected provider=rules”, check relevance thresholds and NeedsLLMRefinement first.

9) Testing Strategy (Backend/API)

Test layers

  • Unit: local logic and component-level behavior under tests/unit.
  • Integration: service-level orchestration + config interactions under tests/integration.
  • E2E: API/frontend contract checks under tests/e2e.

Recommended verification commands

go test ./tests/unit -run 'TestServiceRun|TestLoadConfig|TestEnricher|TestExporter|TestCollectArticles' -count=1
go test ./tests/integration -count=1
go test ./... -count=1

10) Safe Change Checklist

If you change API request/response fields

  • Update handlers and any scheduler/worker callers.
  • Update e2e/API tests that assert payload shape.

If you change relevance/profile thresholds

  • Re-check enricher/provider routing tests.
  • Re-check integration tests expecting provider behavior.

If you change entity matching or entity config semantics

  • Re-run entity mapper + service entity selection tests.
  • Validate unknown/alias/symbol behavior explicitly.

If you change impact/export formatting

  • Re-run parent-child unit tests and CSV export tests.
  • Validate integration export artifacts.

This document is intentionally backend/API-centric and should be updated whenever execution semantics, config contracts, or endpoint behavior changes.
